## Why it matters
The `seen`, `report`/`reportConfirmYes` and `deleteMsgClbk` callbacks parse the
target user (`token`) AND a target message id (`targetMid` / fields 2+) directly
from user-controlled `callback_data`, then perform a privileged action against
the *target's* chat using that message id. The only guard is an `IsBlocked`
check; nothing binds the supplied message id to a message that the caller is
actually entitled to act on. callback_data is echoed verbatim by Telegram, and
the format (`seen|<encoded_chevaletid>|<mid>`) is fully known, so a user who has
ever received one message from a victim can edit the embedded `<mid>` (and reuse
the victim's encoded chevaletid, which never changes) to drive actions against
arbitrary message ids in the victim's chat.

## Evidence
- `internal/bot/callbacks.go:81-122` (`seen`): `token, targetMid := parts[1], parts[2]`,
  resolves `targetUID`, then `tg.SendMessage(tid, txtSeenSent, ReplyParameters{MessageId: mid})`
  where `mid` came straight from `targetMid`. The "you've been seen" notice is sent
  into the victim's chat replying to an attacker-chosen message id.
- `internal/bot/callbacks.go:388-402` (`reportConfirmYes`): `tg.CopyMessage(reportChatID, targetID, targetMid64, ...)`
  copies message `targetMid` out of the *target's* chat (targetID) into the report
  channel. `targetMid` is attacker-supplied.
- `internal/bot/callbacks.go:460-478` (`deleteMsgClbk`): collects message ids from
  *every* button's `callback_data` (fields 2+) and calls
  `tg.DeleteMessage(targetID, mid, nil)` for each, where `targetID` is derived from
  the supplied `token` and the ids are supplied in `callback_data`.

## Proof / abuse case (defensive)
1. Victim V sends an anonymous message to attacker A. A now holds a button
   `seen|<encChid_V>|<mid>`; `<encChid_V>` decodes to V's (long-lived) chevaletid.
2. A crafts callback_data `seen|<encChid_V>|<arbitrary_mid>` and triggers it. The
   bot resolves V via the chevaletid and sends "this message of yours was seen"
   into V's chat as a reply to `<arbitrary_mid>` â€” a notification spoof / nuisance
   referencing messages A never received.
3. For `delete`: because `deleteMsgClbk` harvests ids from all buttons and deletes
   them from `targetID`'s chat, a forged keyboard lets A request deletion of
   chosen message ids in V's chat. (Telegram's bot API only permits deleting the
   bot's own recently-sent messages, which caps real impact to bot-sent messages,
   but the authorization decision is still made entirely from attacker input.)
The actionable defect: the message id is trusted from the client with no server-
side record that this (caller, target, mid) triple corresponds to a real
delivered message.

## Remediation (concrete)
- Do not trust `targetMid` from `callback_data` as an authorization key. Persist a
  server-side mapping of delivered messages: when a message is copied to the
  target (`sendmsg.go:copyToTarget`), store `(copied_message_id_in_target_chat,
  sender_uid, target_uid)`; on `seen`/`report`/`delete`, look the action up by the
  *callback message's own* identity and verify the caller is the legitimate party,
  instead of re-deriving everything from editable callback_data.
- At minimum, validate that the message id being acted on was one the bot actually
  sent in that direction (store sent ids), and reject callbacks whose embedded
  chevaletid/mid pair is not on record.
