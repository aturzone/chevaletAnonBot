## Why it matters
The whole product promise is anonymity, and `SECURITY.md` explicitly scopes "any
way to recover a Telegram user id from a callback_data token without the
database." The `chevaletid` cipher is NOT a secret-bearing cipher: it is a Caesar
shift whose key is **carried inside the token itself** (the "key-patch": a random
letter followed by `ord(letter)+key`). `DecodeChevaletID` recovers the key purely
from the token with no secret input. Therefore anyone holding a token can decode
it to the plaintext chevaletid offline, with zero access to the bot or DB.

Worse, with `key == 0` the ciphertext equals the plaintext verbatim (confirmed by
the golden vectors), so a meaningful fraction (1 in 101) of tokens expose the raw
chevaletid with no transformation at all.

## Evidence
- `internal/encoder/encoder.go:77` `key := rand.IntN(keyMaxInt + 1)` (key space [0,100]).
- `internal/encoder/encoder.go:93-95` the key-patch appends `letter` then
  `strconv.Itoa(int(patch) + key)` — i.e. the key is encoded into the token.
- `internal/encoder/encoder.go:124-141` `DecodeChevaletID` derives `key := n - int(keyPatchLetter)`
  from the token alone and reverses the shift. No secret/HMAC anywhere.
- Golden vectors confirm key=0 is identity: `internal/encoder/testdata/golden.json:3-5`
  `pt:"abcXYZ0189gkMNpq_q"` -> `enc:"abcXYZ0189gkMNpq_qa97"` (chevaletid bytes unchanged).

## Proof / abuse case (defensive)
The plaintext chevaletid is what `GetUIDByChevaletID` looks up to find the real
Telegram uid (`internal/db/users.go:48`). Decoding a token does not by itself
reveal the uid (that mapping lives in the DB), but it collapses the cipher to a
stable per-user identifier:
- **Correlation / linkage:** the same user's encoded tokens all decode to one
  fixed chevaletid, so an attacker can confirm "these N anonymous messages, across
  different chats/links, came from the same person," defeating per-link
  unlinkability. (Encodings differ on the wire due to the random key, which gives
  a false impression of rotation; decoding removes it.)
- **Forgery substrate:** because tokens are self-minting (any plaintext can be
  re-encoded with `EncodeChevaletID`, and any captured chevaletid can be re-shifted
  with a fresh key), an attacker who learns a victim's chevaletid (e.g. via a
  key=0 token, or a leaked DB row, or finding 33h/47h byte patterns) can forge new
  valid tokens for that victim — this is the substrate that makes S-0001/S-0006
  exploitable across messages.

## Remediation (concrete)
- Replace the Caesar/key-patch scheme with an authenticated, keyed token:
  `token = base64url( nonce || AEAD_seal(server_key, nonce, chevaletid || context) )`,
  where `context` binds the token to its purpose and message (target uid + message
  id), using a server-only key from config. Verify/peel on receipt; reject on auth
  failure. This makes tokens both confidential and unforgeable.
- If full re-architecture must wait for after cutover, treat the current scheme as
  plaintext-equivalent in the threat model and DO NOT rely on it for any
  authorization decision (see S-0001).
