## Why it matters
A user fully controls their display `name` and `custom_tag`/`audio_tag`. These are
stored as raw `OriginalHTML()` (the message rendered back into Telegram HTML) and
later interpolated, UNescaped, into messages sent to OTHER users (and to the
sender) with `ParseMode: "HTML"`. Telegram only supports a small HTML subset, but
this still allows injecting attacker-controlled `<a href=...>`, `<b>`, `<code>`,
`<blockquote>`, etc. into a victim's chat — and, more dangerously for an anonymity
product, an `<a href="tg://user?id=...">` or `<a href="http://attacker/...">`
link rendered as if it were part of the bot's trusted message.

## Evidence
- Stored raw HTML:
  - `internal/bot/settings.go:213` `newName := msg.OriginalHTML()` then
    `internal/bot/settings.go:224` `b.DB.SetName(dbctx, userid, newName)`.
  - `internal/bot/settings.go:293` `newTag := msg.OriginalHTML()` then
    `set(dbctx, userid, &newTag)` (custom + audio tags).
  - Length is checked with `runeLen(...) > MaxNameLength` (`settings.go:214`,`:294`)
    which counts code points of the HTML, not visible text — so markup "spends" the
    budget but is still stored.
- Rendered to another user, unescaped:
  - `internal/bot/sendmsg.go:364-370` (`warningHandle`, channel-reply branch):
    `name, _ := b.DB.GetName(...); sentText = fmt.Sprintf("فرستادم به %s.\n<blockquote>...", name, ...)`
    then sent at `sendmsg.go:423` with `ParseMode:"HTML"`. The code comment at
    `sendmsg.go:369` explicitly notes "The name is NOT escaped, matching the original."
  - `internal/bot/sendmsg.go:276-297` (`addTag`) appends the target's `custom_tag`/
    `audio_tag` to the copied message text/caption and edits it with `ParseMode:"HTML"`.
    Here the tag is the *target's* own (self-XSS-ish), but combined with the
    reply/answer flow a sender's identity strings also surface.
  - `internal/bot/start.go:155-159` interpolates `name` into the connect prompt with
    `ParseMode:"HTML"`.

## Proof / abuse case (defensive)
1. Attacker sets their display name via /settings -> change name to:
   `<a href="https://evil.example/track">tap me</a>` (within the rune limit).
2. Victim replies to the attacker's channel post (channel-reply flow). The
   "فرستادم به %s" confirmation rendered in the victim's chat now contains the
   attacker's clickable link, presented inside the bot's trusted message. This is a
   phishing / tracking-link primitive and an integrity break of bot messages; a
   crafted `tg://user?id=` or deceptive link undermines the anonymity UX.
3. Malformed entities (e.g. unbalanced tags) can also trigger Telegram
   "can't parse entities" failures on messages to the victim, a low-grade DoS of
   the delivery path.

## Remediation (concrete)
- Treat names/tags as untrusted display text. Either (a) store and render them
  escaped via `html.EscapeString` at every HTML sink, or (b) restrict the stored
  value to a safe entity subset by re-serializing only allowed tags, and strip
  `href` schemes other than a strict allowlist. The simplest robust fix: store the
  plain `msg.Text` (not OriginalHTML) for names, and `html.EscapeString` before
  interpolation in `warningHandle`/`startConnect`.
- Enforce the length limit on visible text, not on the HTML byte/rune count.
