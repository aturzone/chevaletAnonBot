## Why it matters
For a bot about to serve 10,000+ users there is no per-user or global rate
limiting on any action. Several flows turn one cheap inbound message into outbound
load against arbitrary victims or against the bot's own memory/DB, and the
anonymity property removes the usual social cost of abuse (the abuser is hidden).

## Evidence
- No throttle in the middleware: `internal/bot/prep.go:27-79` runs init + ban check
  then dispatches; there is no rate/quota gate (only a per-user mutex for state
  safety, `prep.go:48-50`).
- **Anonymous spam to a victim:** every `/start <cid>` + message delivers a copy to
  the target (`internal/bot/sendmsg.go:237` `copyToTarget`). A user can send
  unlimited anonymous messages to any victim whose cid they have; the only barrier
  is the victim blocking each sender chevaletid (which rotates per S-0002).
- **Report-bombing:** `internal/bot/callbacks.go:329-409` (`reportConfirmYes`) sends
  a header + copies the reported message into REPORT_CHAT_ID on every confirm, with
  no per-reporter/target cap and no dedupe. Repeated reports flood the moderation
  channel and the reported user's report count.
- **Mass message:** `internal/bot/background.go:253-291` (`sendMassMsg`) loops over
  *all* uids copying with no pacing ("No rate limiting, matching the Python
  original", comment at `background.go:252`). This is admin-gated (see S-0005 path)
  but will hit Telegram global flood limits and can wedge delivery for the whole
  user base.
- **Unbounded memory growth:** `internal/bot/userdata.go:66-82` (`userStore`) creates
  a `*userData` per distinct Telegram user id and NEVER evicts. Any user (incl.
  one-shot spammers) permanently consumes a map entry; over a large/hostile
  population this grows without bound. The conversation `InMemoryStorage`
  (`internal/bot/handlers.go:121,150,194`) is similarly unbounded.

## Proof / abuse case (defensive)
- A single script driving the public `/start <cid>` flow can deliver thousands of
  anonymous messages to a chosen victim per minute, limited only by Telegram's
  per-bot send rate — i.e. the bot becomes an anonymous harassment amplifier.
- Repeated `report_yes` taps on one delivered message flood REPORT_CHAT_ID.
- Churning distinct user ids (or simply growth to 10k+ real users plus abusers)
  steadily expands `userStore`/conversation storage; with no TTL this is a slow
  memory-exhaustion vector.

## Remediation (concrete)
- Add a per-user token-bucket in `prep` (and a separate, stricter bucket for
  outbound-to-victim actions: send_msg, seen, report) keyed by `userid`; reject or
  delay over-limit updates.
- Cap reports per (reporter,target) and per target per time window; dedupe identical
  reports.
- Pace `sendMassMsg` (e.g. <= ~25–30 msg/s with backoff on 429) and make it
  cancellable.
- Bound `userStore` and the conversation storage: use an LRU / TTL eviction so idle
  users are reclaimed.
