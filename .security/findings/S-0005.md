## Why it matters
Internal error detail is reflected to end users. While the central `onError`
returns only a tracking code to the user (good) and pages full detail to
ERROR_CHAT_ID, two paths leak raw error strings directly into the user's chat,
which can disclose database structure, internal identifiers, or stack-ish detail
useful for further attack — and in an anonymity product, error text that includes
ids/usernames is especially sensitive.

## Evidence
- `internal/bot/admin.go:45` (admin default error branch):
  `_ = b.replyText(ctx, fmt.Sprintf("error: %T | %v", err, err))` — sends the Go
  error type and message to the chat. (Admin-only, lower exposure, but still leaks
  internals such as pgx/Postgres error text.)
- `internal/bot/admin.go:101` and `:102` reflect `cerr`/send failures verbatim to
  the admin chat.
- Database errors to ordinary users are handled safely (generic notice + report):
  `internal/bot/prep.go:111-116`. The concern is the non-DB/non-network `default`
  path: `internal/bot/prep.go:122-124` returns the error to `onError`, which replies
  only a code (`internal/bot/bot.go:174-179`) — acceptable. So the principal leak is
  the admin command surface and any handler that itself formats `%v` of an error
  into a user reply.

## Proof / abuse case (defensive)
An admin (or anyone who reaches an error-returning admin subcommand) sees raw
backend errors; if any future non-admin path adopts the same `fmt.Sprintf("...%v", err)`
-> reply pattern, it would disclose internals to arbitrary users. Flagged now to
prevent that pattern spreading and to tighten the admin surface.

## Remediation (concrete)
- Never send `err.Error()`/`%v`/`%T` of an internal error to a user chat. Reply a
  generic message + a tracking code (as `onError` already does) and log the detail
  server-side / to ERROR_CHAT_ID only.
- Replace `admin.go:45` and the `:101-102` reflections with a code + log.
