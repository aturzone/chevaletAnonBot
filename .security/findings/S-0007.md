## Why it matters
The bot makes a server-side POST to `b.Dyn.AIURL()`, a value that is changeable at
runtime via the `/admin ai-url set <url>` command and persisted to
`dynamic_settings.json`. There is no scheme/host allowlist, no block on internal
ranges, and the request runs from the bot host's network position. If an admin
account is compromised (or an attacker reaches the admin path another way), this
is a direct SSRF into internal services (cloud metadata, DB admin ports, etc.).

## Evidence
- `internal/bot/background.go:195-205` builds and sends `http.NewRequestWithContext(..., POST, b.Dyn.AIURL(), ...)`
  via `aiHTTP` (`background.go:27`), a plain client with NO proxy and NO host
  restriction. Note the comment: "NOT via the Telegram proxy".
- `internal/bot/admin.go:188-194` `ai-url set` -> `b.Dyn.SetAIURL(v)` with `v`
  taken straight from the admin's message; no validation.
- `internal/dynset/dynset.go:99` `SetAIURL` stores the value as-is and persists it.
- The response body is parsed for an `output` field and (Cf-stripped) content is
  sent into the GM group (`background.go:173-188`) — a limited exfil/echo channel
  for whatever the URL returns.

## Proof / abuse case (defensive)
With admin access: `/admin ai-url set http://169.254.169.254/latest/meta-data/...`
(or an internal host:port). The bot's AI worker then POSTs to that address from
inside the trust boundary; the response (`output` field) is echoed to the GM group.
Even without a JSON `output` field, the *request itself* reaches internal targets
(blind SSRF: port-scan via timing/error differences). Severity is held at medium
because it requires the admin role; treat as high if admin auth is weak or shared.

## Remediation (concrete)
- Validate `ai-url` on set: require `https://`, resolve the host and reject
  private/loopback/link-local/metadata ranges (deny RFC1918, 127/8, 169.254/16,
  ::1, fc00::/7), and ideally pin to a fixed allowlist of expected AI hosts.
- Re-validate at request time (DNS can change between set and use; resolve + check,
  or use a dialer that blocks internal IPs).
- Consider routing this egress through the same restricted proxy as Telegram, or a
  dedicated egress with no internal reachability.
