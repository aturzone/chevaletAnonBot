## Why it matters
`startUnblock` takes a numeric uid straight from the deep-link parameter and, when
that uid is currently blocked by the caller, replies with the target's `@username`
and a `tg://user?id=` link via `getUsername`/`hrefUser`. The username lookup calls
`GetChat(uid)` on a caller-influenced id. The information returned (current
@username for a given numeric id) is mostly low-sensitivity, but it is an
unauthenticated-by-relationship lookup that confirms id->username for ids the
caller chooses (bounded by the "must already be blocked by you" condition).

## Evidence
- `internal/bot/start.go:42-72` (`startUnblock`): `targetUID = arg[i+1:]` from the
  `UNBLOCK-<uid>` deep link; on the blocked branch it builds
  `b.getUsername(targetUID) + " | " + hrefUser(targetUID, "")` and replies.
- `internal/bot/getuser.go:42-56` (`getUsername`): `b.TG.GetChat(id)` for the
  supplied id and returns `@<username>`.
- The guard is `IsBlocked(userid, targetUID)` (`start.go:60`), so disclosure is
  limited to ids the caller has previously blocked — but blocks are created from
  attacker-supplied chevaletids (see S-0001/S-0002), so the set is not strictly
  self-limiting.

## Proof / abuse case (defensive)
A caller who has (or can induce) a block row for a target uid can retrieve that
target's current @username and a tg:// link by sending `/start UNBLOCK-<uid>`.
Low impact in isolation; noted because it is an authorization decision made on a
client-supplied identifier and contributes to identity correlation.

## Remediation (concrete)
- Do not echo a target's username/profile link for an arbitrary supplied uid.
  Show only an opaque confirmation ("unblocked"), or resolve the display strictly
  from data the caller is entitled to.
- Apply the same to `/admin link` only-if-needed (admin-gated, acceptable), but
  keep the user-facing path opaque.
