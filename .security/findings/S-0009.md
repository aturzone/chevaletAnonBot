## Why it matters
`dynamic_settings.json` can hold the AI session id (a credential-like value passed
as `sessionId` to the AI backend) and is written with mode `0o644` (world-readable
on a multi-tenant host). The mass-message failure file (which contains a full list
of user ids) is likewise `0o644`. On a shared or container-with-extra-mounts host
these are readable by other local users/processes.

## Evidence
- `internal/dynset/dynset.go:63` `os.WriteFile(s.path, b, 0o644)` for
  `dynamic_settings.json` (path set in `internal/bot/bot.go:75`). Contents include
  `ai_session_id` when overridden (`dynset.go:108`).
- `internal/bot/background.go:281` `os.WriteFile(logfile, ..., 0o644)` for
  `mass-msg-failurs.txt` (a list of all uids that failed delivery).
- `.env` itself is read (`internal/config/config.go:218-258`) and git-ignored
  (`.gitignore:6-7`), which is correct; the file-mode concern is the runtime-written
  files above. .env.example uses placeholders only (no real secret committed) —
  verified `.env.example:1-55`.

## Proof / abuse case (defensive)
Any local user/process that can read the working directory obtains the AI session
id and the full user-id list. Not remotely exploitable, but a needless secret/PII
exposure on the host.

## Remediation (concrete)
- Write `dynamic_settings.json` with `0o600`; create the file with restrictive
  perms (use `os.OpenFile(path, O_WRONLY|O_CREATE|O_TRUNC, 0o600)`).
- Write/remove the mass-msg failure file with `0o600`, or keep it in memory and
  send as a document without touching disk.
- Confirm the production secret (BOT_TOKEN, DB_PASS, AI_SESSION_ID) live only in a
  `0600` .env / secrets manager and are rotated if they were ever committed or
  shared. (No secret value is reproduced here.)
