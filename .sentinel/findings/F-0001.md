## Why it matters
`Bot.Run` returns immediately after `Updater.Stop()` (bot.go:122), and `main` then runs
`defer database.Close()` (main.go:45) and exits. `Updater.Stop()` waits for in-flight *dispatcher*
handlers (verified: ext/dispatcher.go:213 `waitGroup.Wait()`), but it does NOT wait for:
- the `time.AfterFunc` one-shots: `scheduleMassMsg` (7s, jobs.go:247), `scheduleDeleteMessage`
  (2s, jobs.go:46), `scheduleDeleteWarning` (10-15s, jobs.go:27);
- the background goroutines started in `startBackground` (background.go:30) after `<-ctx.Done()`.

`sendMassMsg` (background.go:253) acquires its own `context.Background()`-rooted dbctx and calls
`b.DB.GetAllUIDs` AFTER the 7s delay. If an admin runs `/admin send-mass-msg YES` and a SIGTERM
arrives within the 7s window (or while the per-user copy loop over 10k users is running), the pool
is closed under it: `GetAllUIDs`/`Copy` fail mid-broadcast, the failure file is written, and the
broadcast is silently truncated. For a 10k-user cutover this means a partial mass-send with no clean
record of who was reached. The auto-delete timers firing post-exit is cosmetic, but the mass-msg
case is real data-impacting work lost on a normal restart.

## Evidence
- internal/bot/bot.go:120-122 ŌĆö `<-ctx.Done(); return b.Updater.Stop()` with no wait for timers/goroutines.
- internal/bot/jobs.go:246-247 ŌĆö `time.AfterFunc(7*time.Second, func() { b.sendMassMsg(msg) })`, untracked.
- internal/bot/background.go:259-261 ŌĆö `sendMassMsg` opens a fresh `context.WithTimeout(context.Background(), ŌĆ”)`, not tied to the run ctx.
- cmd/bot/main.go:45 ŌĆö `defer database.Close()` runs as soon as `b.Run` returns.
- ext/dispatcher.go:212-218 ŌĆö `Stop()` only waits on the dispatcher waitGroup (timers/goroutines are not registered there).

## Suggested action
Have `Bot.Run` own a `sync.WaitGroup` for background goroutines and await it after `Updater.Stop()`,
and gate the auto-delete/mass-msg helpers on a shutdown signal. Concretely: store the run `ctx` on
`Bot`, register the background loops on a WaitGroup, and in `Run` do `b.Updater.Stop(); b.bgWG.Wait()`
before returning. For `sendMassMsg`, derive its dbctx from the run ctx (not `context.Background()`)
so an in-flight broadcast aborts deterministically, and persist progress so it is resumable. At
minimum, document that a restart during a mass-send truncates it.
