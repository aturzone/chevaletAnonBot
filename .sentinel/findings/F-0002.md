## Why it matters
The design invariant (userdata.go:16-21, prep.go:45-50) is "two updates from the SAME user never
race because prep holds `ud.mu` for the whole handler." That lock protects the `convData` struct,
but it does NOT protect the gotgbot ConversationHandler's own state machine. The dispatcher runs up
to 50 updates concurrently (ext/dispatcher.go:20, no MaxRoutines set in bot.go:85-87). For one user
the sequence per update is:

1. `Conversation.CheckUpdate` -> `StateStorage.Get` reads current state  (BEFORE prep's lock)
2. the matched handler runs under `ud.mu`  (prep.go:49)
3. `Conversation.HandleUpdate` -> `StateStorage.Set/Delete` writes the new state  (AFTER prep's lock)

Steps 1 and 3 are outside `ud.mu`. The `InMemoryStorage` is internally locked per call
(conversation/in_memory.go:36,56,73) but the read-decide-write across an update is not atomic w.r.t.
the per-user mutex. If a user fires two messages within milliseconds (e.g. an auto-reply followed
immediately by a manual send, or fast media), both goroutines can read the same `currState`, both
select a state-0 `sendMsg` handler, and run serialized via `ud.mu` but with already-stale routing —
producing a duplicated/misrouted send rather than one composing one delivering. It is not memory
corruption (the mutex prevents that), but it is a logic race on the conversation step that the
per-user lock was believed to cover.

## Evidence
- internal/bot/prep.go:48-50 — `ud.mu.Lock()` is taken inside the handler body, i.e. between the conversation's Get (CheckUpdate) and Set (after HandleUpdate).
- ext/handlers/conversation.go:192 — `currState, err := c.StateStorage.Get(ctx)` in `getNextHandler`, called from CheckUpdate before the handler runs.
- ext/handlers/conversation.go:110,122 — `StateStorage.Delete` / `Set` happen after `next.HandleUpdate` returns.
- ext/dispatcher.go:20 + internal/bot/bot.go:85-87 — DefaultMaxRoutines=50, dispatcher created without MaxRoutines, so per-user updates can be in two goroutines at once.

## Suggested action
Make the conversation transition atomic with the per-user critical section. The cleanest fix without
editing the library: in `prep`, take `ud.mu` and keep the conversation single-flight per user by
serialising at the dispatcher boundary for that user (e.g. a per-user "in flight" guard that defers
the second update). Alternatively, accept and document the window (the worst case is a rare duplicate
delivery, deduped by the user) — but it should be a conscious decision, since the code comments
currently assert the lock fully serialises a user's updates.
