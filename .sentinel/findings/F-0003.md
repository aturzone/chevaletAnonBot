## Why it matters
`scheduleMassMsg` stores `msg := ctx.EffectiveMessage` and then `time.AfterFunc(7s, …)` uses it.
That capture is of the *message pointer*, which is fine in isolation, but the surrounding pattern is
fragile: `*ext.Context` from gotgbot is created per update (ext/dispatcher.go:257 `NewContext`) and
is not safe to retain past the handler — its fields (Data map, etc.) may be mutated/reused by the
framework. Here only `EffectiveMessage` (a decoded value pointer) is captured, so today it happens to
be safe, but the same file also passes `ctx *ext.Context` directly into a deferred 7s closure via
`b.scheduleMassMsg(ctx)` at admin.go:82 -> the closure could be one refactor away from touching a
stale `ctx`. Combined with F-0001 (the goroutine outliving the run ctx and the DB pool), this is the
riskiest deferred-work path in the cutover.

## Evidence
- internal/bot/background.go:245-248 — `func (b *Bot) scheduleMassMsg(ctx *ext.Context) { msg := ctx.EffectiveMessage; time.AfterFunc(7*time.Second, func() { b.sendMassMsg(msg) }) }`.
- internal/bot/admin.go:82 — `b.scheduleMassMsg(ctx)` passes the live update context into the scheduler.
- ext/dispatcher.go:256-257 — `ProcessUpdate` builds a fresh `ctx := NewContext(...)` per update; nothing keeps it alive for the handler's spawned goroutines.

## Suggested action
Extract everything the deferred broadcast needs into plain values at schedule time (the replied-to
`*gotgbot.Message` and the admin chat id) and pass only those into the closure — never the
`*ext.Context`. Pair with F-0001's run-ctx + WaitGroup so the broadcast is cancellable and awaited.
