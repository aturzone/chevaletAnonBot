## Why it matters
`userStore.get` lazily creates a `*userData{mu, convData}` for every distinct Telegram user id that
ever sends an update and stores it in `s.m`, but nothing ever deletes entries. With 10k+ users (and
growing) hitting the bot over a long-lived single process, this is a permanent, monotonically
growing map. Each entry is small (a mutex + the convData struct of mostly empty strings), so this is
not an urgent leak — at 10k users it is on the order of a few MB — but it never plateaus and there is
no upper bound, so over months of uptime with new users it only goes up. For a process expected to
run unattended after the cutover, an unbounded in-memory structure keyed by an external id is worth
bounding.

## Evidence
- internal/bot/userdata.go:73-82 — `get` does get-or-create into `s.m[uid]`; no eviction anywhere.
- internal/bot/userdata.go:66-69 — `userStore{ mu, m map[int64]*userData }`, only `m[uid]=` and reads, confirmed by grep (no `delete(s.m, …)` exists).

## Suggested action
Evict idle `userData` whose `convData` is empty (the common steady state). Options: (a) when a
handler ends a conversation and `u.d == convData{}`, drop the entry from the store under `s.mu`; or
(b) a periodic sweep that removes entries whose data is zero-valued and that are not currently locked.
Because the per-user mutex must not be discarded while held, gate eviction on a successful non-blocking
`TryLock` (Go 1.18+) and only delete zero-valued, unlocked entries. Low priority, but it removes an
unbounded growth path from a long-running process.
