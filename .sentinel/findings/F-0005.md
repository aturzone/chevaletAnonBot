## Why it matters
This is the only notable spot in an otherwise clean resource-handling picture (pgx rows are closed
everywhere — db/reports.go:47 `defer rows.Close()` + `rows.Err()`, and db/cids.go:18 /
db/users.go:39 use `pgx.CollectRows` which closes internally; file handles in adminBackup.go:309 and
here are deferred/closed). In `sendMassMsg` the failure-log path writes `mass-msg-failurs.txt`, opens
it, sends it as a document, then unconditionally `os.Remove`s it. If `ReplyDocument` errors (its
result is discarded with `_`), the temp file is still removed, so the admin gets neither the document
nor the on-disk record — the list of which of the 10k users failed is lost. Also the working
directory is assumed writable (fine in the container, but undocumented).

## Evidence
- internal/bot/background.go:281-287 — `os.WriteFile(logfile, …); os.Open(logfile); _, _ = msg.ReplyDocument(...); f.Close(); _ = os.Remove(logfile)`; the ReplyDocument error is dropped.
- internal/bot/background.go:274-276 — per-user copy failures are accumulated into `failures` (good), but only surfaced via that one best-effort document.

## Suggested action
Check the `ReplyDocument` error; only `os.Remove` the file on a successful send, and log a warning
(with the path) when the send fails so the record survives for manual retrieval. Consider writing to
a stable, mounted path rather than the CWD.
