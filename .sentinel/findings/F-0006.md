## Why it matters
Good news for the cutover question "panics that could crash the single bot process": they do NOT
crash it. gotgbot recovers every handler panic per-update (ext/dispatcher.go:259-272), and prep's
`defer ud.mu.Unlock()` (prep.go:50) still runs during unwinding, so a panic does not deadlock the
per-user lock either. That risk is clean.

However, the dispatcher is built with only `Error: b.onError` and no `Panic` handler
(bot.go:85-87). When `Panic` is nil, a recovered panic is turned into an error returned from
`ProcessUpdate` and merely logged by the dispatcher's default logger (ext/dispatcher.go:266-270,
204) — it is NOT routed through `onError`. Consequences: (1) the user who triggered the panic gets
no "خطایی رخ داد / tracking code" reply (unlike every normal error via onError, bot.go:174-179);
(2) ERROR_CHAT_ID receives no report, so operators are blind to nil-deref/index panics post-cutover
unless they are tailing container logs. Several handlers index slices after only partial length
checks (e.g. sendmsg.go:389-391 `parts[1:]` after splitting callback data; media.go:99 `msgs[0]`),
so a panic is plausible on malformed/edge input.

## Evidence
- internal/bot/bot.go:85-87 — `ext.NewDispatcher(&ext.DispatcherOpts{Error: b.onError})`; no `Panic:` field.
- ext/dispatcher.go:259-271 — recover(): if `d.Panic == nil`, the panic becomes a returned error, NOT a call into the Error handler.
- ext/dispatcher.go:200-206 — that returned error is sent to `UnhandledErrFunc`/`Logger` only.
- internal/bot/bot.go:141-181 — `onError` (the only place that DMs the user a tracking code and reports to ERROR_CHAT) is the dispatcher *Error* hook, which panics never reach.

## Suggested action
Set `Panic:` on the DispatcherOpts to a handler that does what onError does for panics: generate a
CID, log with the stack, report a compact summary to ERROR_CHAT_ID, and reply the tracking code to
`ctx.EffectiveMessage`. A thin adapter `func(b, ctx, r any){ b.onError(b.TG, ctx, fmt.Errorf("panic: %v", r)) }`
(plus the stack in the log) restores parity so operators see panics during the cutover.
