## Why it matters
`prep` runs `initUser` (a DB upsert + several SELECTs) and `IsBanned` on EVERY update before the real
handler (prep.go:55-69). If PostgreSQL is briefly unavailable — exactly the kind of transient blip
expected around a shared-DB cutover — every in-flight update for every active user takes the
`isDBError` branch, which sends "PostgreSQL ERROR: …" to ERROR_CHAT_ID with no rate limiting or
dedupe (prep.go:111-116, reportToErrorChat at bot.go:186-195). With 10k users this can mean hundreds
of identical messages to the error channel within seconds, which (a) can itself hit Telegram's
per-chat send limits and get the bot 429'd, and (b) buries operators in noise at the worst possible
moment. The Python original had the same per-handler report, but the Go port's higher concurrency
(50 simultaneous updates) makes the burst sharper.

## Evidence
- internal/bot/prep.go:55-69 — `initUser` + `IsBanned` run for every update inside prep, so a DB outage triggers the error path for every update.
- internal/bot/prep.go:111-116 — `case isDBError(err): … b.reportToErrorChat(tg, "PostgreSQL ERROR: "+err.Error())` per occurrence.
- internal/bot/bot.go:186-195 — `reportToErrorChat` is an unconditional best-effort `SendMessage`, no throttle/coalescing.
- internal/bot/tgerr.go:65-72 — `isDBError` matches both PgError and pgconn.ConnectError, so connection-refused during an outage routes here for all updates.

## Suggested action
Coalesce DB-error reports: keep a small "last DB-error report at T" timestamp on `Bot` and only post
to ERROR_CHAT_ID at most once per N seconds (e.g. 30s) while errors persist, with a count
("PostgreSQL errors: 412 in the last 30s"). The user-facing "database problem" reply can stay
per-update. This keeps signal without the flood and avoids self-inflicted 429s during a cutover blip.
