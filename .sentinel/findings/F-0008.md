## Why it matters
At boot, `db.Connect` does a `pool.Ping` (db.go:72-75) and any failure makes `main` log and
`os.Exit(1)` (main.go:41-44). The same is true for `MakeTables` (main.go:46-49). During the cutover
the Go bot is started right after the Python bot is stopped while sharing `telegram-bot-db`; if the
DB has a momentary hiccup at exactly that instant, the Go container exits.

Mitigation already in place (verified): the prod compose sets `restart: unless-stopped` and
`init: true` (docker-compose.prod.yml:27-28), so the container self-heals by restarting until the DB
answers. There is no `depends_on` ordering — and there can't be, because telegram-bot-db is an
`external` container, not a service in this compose (networks.prod.external: true, line 75-76). So
the crash-restart loop IS the ordering mechanism. The residual risk is narrow: during the loop the
bot is not polling, so the cutover gap widens (Telegram buffers updates up to 24h, so nothing is
lost), and the runbook's step-3 "watch for `bot polling`" is how an operator would notice it is
looping. pgxpool tolerates a DB that drops *after* a successful start, so this is purely a
startup-window concern.

## Evidence
- internal/db/db.go:72-75 — `if err := pool.Ping(ctx); err != nil { pool.Close(); return … }`.
- cmd/bot/main.go:40-49 — both `db.Connect` and `database.MakeTables` failures call `os.Exit(1)`.
- deploy/go/docker-compose.prod.yml:27-28 — `restart: unless-stopped`, `init: true` (self-heal present).
- deploy/go/docker-compose.prod.yml:74-80 — DB is an external network/container, so no `depends_on` gate is possible.

## Suggested action
Optional hardening: add a bounded connect-retry loop at startup (retry Ping for ~30s with backoff
before `os.Exit`) so a momentary blip during cutover doesn't widen the polling gap with a restart.
Otherwise, this is acceptable as-is given `restart: unless-stopped`; just keep the runbook check for
`bot polling` (and confirm the container is not in a restart loop) as an explicit cutover sign-off
item.
