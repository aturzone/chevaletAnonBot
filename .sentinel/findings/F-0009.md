## Why it matters
`internal/bot` has ZERO tests (verified: only internal/db — live-PG, skipped without DB_HOST — and
internal/encoder have tests). The single most fragile pure-logic routine in the package is the
deletion-button packer: `warningHandle` splits `deletionCallbackData` into `encChid` + message ids,
then greedily packs ids into as many "delete|<encChid>|<mid>|<mid>…" buttons as fit under Telegram's
64-byte callback_data limit (sendmsg.go:388-417); `deleteMsgClbk` later reverses this by reading
fields 2+ from EVERY button to collect the ids to delete (callbacks.go:460-478). If the packing
miscounts (the `isValid` length check at sendmsg.go:393 uses `len(strings.Join(rm,"|")) <= 64` on
the field slice — a byte count that must exactly match what Telegram enforces), a button's
callback_data can silently exceed 64 bytes and Telegram rejects the whole `sendMessage`, OR an id can
be dropped/duplicated across buttons and the wrong messages get deleted. For an anonymous-message bot
the "delete the message I sent to my contact" button is a core trust feature — getting it wrong
post-cutover deletes the wrong message or fails to delete at all. The media-group path
(media.go:176-185) builds the same callback data with potentially many ids, exercising the multi-button
split that is hardest to get right.

## Evidence
- internal/bot/sendmsg.go:388-417 — the pack loop: `kbTemplate`, `isValid`, the `for _, mid := range mids` greedy fill, and the `equalStringSlices` flush of the trailing chunk.
- internal/bot/callbacks.go:460-478 — the unpack: iterate keyboard, `strings.Split(btn.CallbackData,"|")`, take `flds[2:]` into a dedup set, delete each.
- internal/bot/media.go:176-185 — builds `deletionCallbackData = encodedTargetChid + "|"+m …` over `dedupeOrdered(sentMedias + notify)`, the many-ids case.
- (No `*_test.go` exists anywhere under internal/bot — grep confirms.)

## Suggested action
Add a table-driven unit test for the pack/unpack round-trip: for N message ids (N = 0,1,2,5,20) and
a representative `encChid`, assert (a) every produced button's `len(callbackData) <= 64`, (b) the
union of ids parsed back out by the deleteMsgClbk logic equals the input set exactly (no drops, no
dupes), and (c) the row chunking (2 buttons/row, sendmsg.go:410-417) is well-formed. Factor the pack
and unpack into small pure functions so they can be tested without gotgbot.
