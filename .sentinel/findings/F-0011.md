## Why it matters
`reportConfirmYes` parses the report channel id with
`reportChatID, perr := strconv.ParseInt(b.Cfg.ReportChatID, 10, 64)` and returns `perr` on failure
(callbacks.go:362-365). But `REPORT_CHAT_ID` is OPTIONAL in config (config.go:109,
`opt("REPORT_CHAT_ID", "")`), so if it is unset or set to a non-numeric value, every user who
confirms a report triggers a parse error -> routed to onError -> the reporting user gets a generic
"خطایی رخ داد / tracking code" reply, and the report is never delivered anywhere. Contrast the
ERROR_CHAT_ID parse sites, which all defensively guard with `perr == nil` (bot.go:154,
background.go:165) — the report path does not. For a moderation feature on a 10k-user anonymous bot,
silently dropping reports (while telling the reporter something errored) is a meaningful safety gap,
and it depends entirely on an operator having set one optional env var correctly at cutover.

## Evidence
- internal/bot/callbacks.go:362-365 — `reportChatID, perr := strconv.ParseInt(b.Cfg.ReportChatID, 10, 64); if perr != nil { return perr }`.
- internal/config/config.go:109 — `c.ReportChatID = opt("REPORT_CHAT_ID", "")` (optional, no validation, defaults to empty).
- internal/bot/bot.go:154 — by contrast ErrorChatID is parsed with `if … perr == nil` (graceful when unset).

## Suggested action
Validate REPORT_CHAT_ID at startup if the report feature is meant to be on (fail fast in config.Load
with a clear message), OR handle the unset case gracefully in reportConfirmYes: if ReportChatID is
empty/invalid, reply a clear "reporting is not configured" message to the user (and log a warning)
rather than emitting a tracking code. Decide whether reporting is mandatory for prod and enforce it
in one place.
