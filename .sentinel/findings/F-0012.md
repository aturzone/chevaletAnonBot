## Why it matters
The chunking loop flushes the accumulated `parts` to the admin when the joined string exceeds 3900
chars, but it never clears `parts` after flushing. So once the threshold is crossed, EVERY subsequent
iteration re-joins and re-sends the entire (still-growing) list, and the final post-loop send
(admin.go:276-278) sends the whole thing yet again. For a report table with enough entries to exceed
3900 chars this produces a burst of large, increasingly-duplicated messages — wasteful and likely to
trip Telegram's per-chat flood limits, getting the admin session 429'd. The intent (and the comment
"flush a chunk … then send whatever remains") is clearly to send each chunk once; the reset was
dropped. Admin-only, so blast radius is small, but it is a clear correctness bug and a plausible
port regression that a structural parity diff would not catch (the Python loop presumably reset its
accumulator after each send).

## Evidence
- internal/bot/admin.go:266-278 — `var parts []string; for … { if len(strings.Join(parts,sep))>3900 { replyText(joined) }; parts = append(parts, …) }; if len(parts)>0 { replyText(strings.Join(parts,sep)) }` — no `parts = nil` after the flush.

## Suggested action
After a successful flush, reset the buffer: `parts = nil` (or `parts = parts[:0]`) immediately after
`replyText(joined)` so each chunk is sent exactly once. Add a small unit test feeding GetAllReports a
map large enough to span 3 chunks and asserting the number/contents of sends.
