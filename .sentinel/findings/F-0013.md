## Why it matters
`config.ExpireAfter = 0.3` seconds (config.go:29). The first album item sets
`groupExpiration = nowSeconds() + 0.3` (sendmsg.go:221), and a later item is rejected when
`nowSeconds() - groupExpiration >= 0.3` (media.go:43), i.e. when it arrives more than ~0.6s after the
first item. Telegram delivers album items as separate updates; normally they arrive within tens of
ms, but the Go dispatcher processes up to 50 updates concurrently and prep serialises a single user's
updates behind `ud.mu` plus a DB upsert/queries per item — so under load, network jitter, or a slow
DB the gap between consecutive album items can exceed 0.6s. When it does, the later items are dropped
with "دیر شد. توی یه پیام جدید بفرست" (txtTooLate) and the album is delivered to the contact
incomplete. This is a faithful port of the Python value, so it is parity-correct, but 0.3s is an
aggressive window for a serialised-per-user + per-item-DB-work pipeline, and the failure mode
(silently truncated album) is exactly the kind of thing that will surface only at 10k-user load, not
in manual testing.

## Evidence
- internal/config/config.go:29 — `ExpireAfter = 0.3 // seconds`.
- internal/bot/sendmsg.go:221 — first item: `ud.d.groupExpiration = nowSeconds() + config.ExpireAfter`.
- internal/bot/media.go:43 — later item: `if nowSeconds()-ud.d.groupExpiration >= config.ExpireAfter { … txtTooLate }`.
- internal/bot/prep.go:48-55 — each item holds `ud.mu` and does `initUser` (a write + reads) before reaching handleMedia, adding per-item latency between album items of the same user.

## Suggested action
Confirm this matches observed Python behavior under real load before cutover; if albums truncate,
raise the window (e.g. ExpireAfter to 1-2s) — the value exists precisely to bound how long the bot
waits for the rest of an album, and a slightly longer wait is harmless. Consider measuring the actual
inter-item gap in logs during the real-data test (CUTOVER.md appendix).
