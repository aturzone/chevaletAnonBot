## Why it matters
`cidTaken` calls `GetAllCIDs` (SELECT cid FROM cids) and `GetAllChevaletIDs` (SELECT chevaletid FROM
users WHERE chevaletid IS NOT NULL) and linear-scans both slices with `indexOf` (mylinks.go:336-349,
db/cids.go:28, db/users.go:59). It runs on every `/my_links` rename attempt (updateCid,
mylinks.go:295). With 10k+ users — each potentially with several cids plus a chevaletid — this pulls
tens of thousands of rows over the wire and allocates them, just to test membership of one candidate
string, every time someone renames a link. The correctness is already guaranteed independently by the
UNIQUE constraint on cids.cid (db.go:115) and the IsUniqueViolation handling in updateCid
(mylinks.go:307-313), so this scan is redundant work whose cost grows linearly with the user base.
It is not a hot path (renames are infrequent), hence low severity, but it is the one place where a
per-action cost scales with total users, which is precisely what changes at 10k.

## Evidence
- internal/bot/mylinks.go:336-349 — `cidTaken`: `GetAllCIDs` + `indexOf`, then `GetAllChevaletIDs` + `indexOf`.
- internal/bot/mylinks.go:295 — called from `updateCid` on every rename.
- internal/db/cids.go:27-30 / internal/db/users.go:57-61 — both are unbounded full-column SELECTs.
- internal/db/db.go:112-115 + internal/bot/mylinks.go:306-313 — the UNIQUE(cid) constraint + IsUniqueViolation already enforce uniqueness, making the pre-check redundant for correctness.

## Suggested action
Replace `cidTaken` with two indexed existence queries (`SELECT 1 FROM cids WHERE cid=$1` and
`SELECT 1 FROM users WHERE chevaletid=$1`, each `LIMIT 1`) — O(1) via the unique index — or drop the
pre-check entirely and rely on the existing IsUniqueViolation path (mylinks.go:307), keeping only the
chevaletid collision check (the cids/chevaletids namespaces are shared, so the chevaletid check is the
only part the cids unique index doesn't cover). Add the chevaletid existence query as an indexed
lookup too.
